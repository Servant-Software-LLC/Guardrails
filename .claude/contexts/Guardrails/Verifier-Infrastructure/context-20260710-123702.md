  --- #350 specialized deterministic guardrail-script LIBRARY (the arc) ---
  - Filed #350. Seed: default TDD-red guardrails only assert `dotnet test` exits non-zero — they can't
    tell all N new tests fail vs just one (a partial red with tautological tests ships green). The
    pilot-seat plan's own `02-tests-fail-on-stubs.ps1` exhibits this (left as a deliberate probe for
    /guardrails-review — see NEXT STEPS).
  - Two determinism refinements (in the issue): (a) parse STRUCTURED results (a TRX logger), NOT
    `dotnet test` stdout — stdout-shape parsing is the #248 brittleness; (b) DERIVE the count (count
    [Fact]/[Theory] in the file, or delta), don't only trust an agent-emitted JSON count (gameable).
  - Bigger arc: LLM re-authors each guardrail `.ps1` per generation → drift/weakness (#248 backwards
    regex, #187 regressed, the whole #346 premise). Fix: ship a `guardrail-lib/` of vetted,
    parameterized scripts the breakdown REFERENCES (`--project`/`--filter`/`--expect-count`) — the
    proactive form of #346's reactive registry.
  - Maintainer pushback captured (comment): a naive count-all-[Fact] breaks in BROWNFIELD (a task
    adding to an existing class with pre-existing passing tests → TRX shows new=failed + old=passed →
    `failed < declared` → false-fails a correct task). Resolution: for the RED case, git
    self-reconstructs the delta — at guardrail time `HEAD == taskBase` (SSOT §6), so
    `git show HEAD:<file>` / `git diff HEAD` gives the pre-task baseline; assert only the NEW test
    names fail. So RED collapses to a SINGLE guardrail (no preflight, retry-safe). The preflight/
    guardrail PAIR is only needed for the GREEN outcome-baseline ("no baseline-green test regressed"),
    which needs a real run at taskBase → the #351 handoff. [Theory] expands to N TRX cases (prefix-match).

  --- #351 verifier store (the enabler) ---
  - Filed #351. Consulted guardrails-architect (Opus) — VERDICT: warranted as a small FOUNDATIONAL
    child-process contract → design/ratify now, implement DEMAND-DRIVEN (wire when #350-green pulls it).
  - Design: a per-task "verifier store" at `logs/<runId>/<task-id>/verifier-store/` — WRITTEN by
    deterministic (script) task preflights (env `GUARDRAILS_VERIFIER_OUT`, once at taskBase before the

    (agent-authored, gameable) — trust class legible by which env var carries it; only deterministic
    verifiers may WRITE. The action gets NEITHER var.

    `logs/` tree (gitignored via `/logs/` in PlanGitignore.cs line 46, outside the segment worktree) so
    the F2 rollback (`git reset --hard <taskBase>` + `git clean -fd`, GitWorktreeProvider.ResetSegment
    ~1187) can't wipe it. My earlier `.guardrails-agent-io/` suggestion was WRONG (inside the segment,
    not gitignored). Cleared on `--fresh` with logs/ (no new teardown).
  - Why preflight-only writer is COMPLETE not under-built: same-attempt guardrail→guardrail already has
    per-attempt `GUARDRAILS_LOG_DIR` scratch; cross-attempt guardrail handoff is UNSOUND (action
